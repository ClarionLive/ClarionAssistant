# LSP server sync (GitHub #40)

Keeps the **bundled Clarion LSP server** anchored to a *tagged* release of
[`msarson/Clarion-Extension`](https://github.com/msarson/Clarion-Extension) instead of a
hand-maintained snapshot that silently drifts from upstream.

## Background

`deploy.ps1` copies the LSP server into the add-in from `$CLARIONLSP_ROOT` — a **git clone of
`msarson/Clarion-Extension`** with our **CodeGraph overlay** applied on top:

| Piece | Tracked upstream? | Needs re-applying on re-pin? |
|-------|-------------------|------------------------------|
| `server/src/codegraph-bridge.ts`, `codegraph-indexer.ts` (our overlay) | **No** — untracked in the clone | **No** — survives `git checkout <tag>` untouched |
| `out/**` (compiled server.js + overlay .js) | **No** — gitignored build output | Regenerated by `npm run compile` |
| Wiring inside `server/src/server.ts` (imports + init + CodeGraph fallbacks + handlers) | **Yes** — tracked, carried as uncommitted edits | **Yes** — must merge onto the new tag's `server.ts` |

So the only genuinely manual step when moving to a new tag is re-merging the **`server.ts`
wiring**. Everything else is mechanical, and `Sync-LspServer.ps1` does it.

## Drift as of 2026-07-03

The deployed snapshot is built from `feat/lsp-member-completion` (`383ef95`, **untagged**),
a branch that forked near the 0.8.7 line — **318 commits behind the `v0.9.6` tag** with 2
experimental commits on top. That's the drift #40 exists to fix. See `lsp-snapshot.json`.

## Usage

```powershell
# Dry run — fetch tags and report how far the bundle has drifted from the target tag
.\lsp-server-sync\Sync-LspServer.ps1

# Re-pin to the target tag (default: targetPin.tag in lsp-snapshot.json) and rebuild
.\lsp-server-sync\Sync-LspServer.ps1 -Apply

# Pin to a specific tag
.\lsp-server-sync\Sync-LspServer.ps1 -Apply -Tag v0.9.6

# After resolving a wiring merge conflict by hand, finish the rebuild + verify
.\lsp-server-sync\Sync-LspServer.ps1 -Apply -SkipBuild
```

Without `-Apply` the script is **read-only** (fetch + drift report only). With `-Apply` it:

1. Refuses if the clone has tracked changes beyond `server.ts` (won't clobber your work).
2. Confirms the CodeGraph overlay `.ts` files are present.
3. Stashes the `server.ts` wiring, checks out the tag, pops the wiring back on.
   - A **pop conflict** is the one step it can't automate — resolve `server.ts` by hand,
     then re-run with `-SkipBuild`.
4. Runs `npm ci && npm run compile`.
5. Verifies the CodeGraph markers survived into the rebuilt `server.js`.
6. Writes the resolved commit + sync date back into `lsp-snapshot.json`.

Then run `deploy.ps1` as usual to copy the freshly-built server into the add-in folder.

## One-time hardening (optional, recommended)

Right now the `server.ts` wiring lives only as uncommitted edits in `$CLARIONLSP_ROOT`. To make
it a durable, re-appliable artifact (so a fresh clone can reproduce the overlay), capture it once
against the pinned tag:

```powershell
cd $env:CLARIONLSP_ROOT
git diff v0.9.6 -- server/src/server.ts > <repo>\lsp-server-sync\codegraph-server-wiring.patch
```

Once that patch exists and is committed here, the sync can `git apply` it onto the tag instead of
relying on stash/pop — fully reproducible. Generating it cleanly requires first rebasing the
overlay off the experimental `feat/lsp-member-completion` branch onto the tag, which is a
coordination point with @msarson (see #40 / #51) — hence it's left as a documented follow-up
rather than auto-generated.

## Coordinating the tag cadence

Pinning to `v0.9.6` pulls the bundle forward 318 commits but drops the experimental
member-completion prototype (`383ef95`). Per #51 the equivalent capability ships in the public
0.9.6 extension. If that prototype must stay on the bundle, ask @msarson to cut a tag at/after
`383ef95` and set it as `targetPin.tag` in `lsp-snapshot.json`.
